# 微信旅游地点打卡小程序设计

设计一个面向单城市或单景区场景的微信小程序。用户到达预置景点附近后才可完成打卡，系统记录旅行轨迹并点亮对应图鉴，提供“记录 + 收集感”的核心体验。

## 产品定义

这是一个旅游场景下的微信小程序：

- 用户授权定位后查看附近可探索的预置景点
- 到达景点附近后才能执行正式打卡
- 打卡后生成个人记录，并点亮景点图鉴
- 首版强调可用的打卡闭环，不做社交社区

## 目标

1. 让用户在真实到达景点附近时完成可信打卡
2. 让每次打卡都沉淀为可回看的旅行记录
3. 通过图鉴、进度、徽章提供轻量收集反馈
4. 用最小范围验证产品价值：先做单城市或单景区

## 非目标

首版不包含以下能力：

- 社交分享、好友可见、排行榜
- 用户自建景点
- 多城市切换或全国景点覆盖
- 评论区、聊天、组队
- 复杂勋章体系
- 路线规划与导航

## 推荐产品方向

在“纯打卡相册型”“地图探索型”“景点图鉴型”三个方向中，首版选择“景点图鉴型”。

原因：

- 比地图导向方案更容易实现稳定的 MVP
- 比纯记录方案更容易形成持续使用的收集动机
- 页面结构清晰，适合先验证打卡闭环

## 用户范围与场景

### 首版范围

- 场景：单城市或单景区
- 地点来源：官方预置景点库
- 打卡方式：基于定位距离判断，用户手动点击打卡按钮完成

### 典型用户流程

1. 用户打开小程序并授权定位
2. 首页展示当前城市/景区的整体进度和附近推荐景点
3. 用户进入图鉴页查看全部景点状态
4. 用户到达某景点附近后进入详情页
5. 系统判断用户在可打卡半径内，按钮切换为“立即打卡”
6. 用户提交打卡，可选上传照片和文字
7. 打卡成功后，图鉴点亮、进度更新、可触发徽章反馈
8. 用户在“我的”页查看自己的旅行时间线与收集进度

## 信息架构

首版建议包含 4 个核心页面。

### 1. 首页

展示内容：

- 当前城市/景区
- 已打卡数量 / 总数量
- 今日或当前最近可探索的景点
- 前往图鉴页的主入口

目标：让用户一眼看到当前进度与最近行动点。

### 2. 景点图鉴页

以卡片列表展示全部预置景点，每张卡片展示：

- 景点封面
- 景点名称
- 距离
- 当前状态

状态分为：

- `未解锁`
- `附近可打卡`
- `已打卡`

支持按状态筛选，帮助用户快速找到下一步目标。

### 3. 景点详情页

展示：

- 景点介绍
- 景点图片
- 与用户当前位置的距离
- 打卡规则与可打卡半径

交互：

- 不在范围内：显示“距离目标还有 xx 米”
- 在范围内：主按钮显示“立即打卡”
- 打卡时允许附加照片和简短文字

### 4. 我的页面

展示：

- 累计打卡数
- 已点亮景点
- 当前完成度
- 已获得徽章
- 打卡记录时间线

## 核心功能设计

### 定位判定

系统在前台页面获取用户当前位置，并用景点坐标计算距离。每个景点配置独立的 `radiusMeters`，默认可在 100m~300m 范围内调节。

设计原则：

- 规则简单明确，用户可理解
- 对 GPS 漂移做适度容错
- 避免用户明明到了现场却无法打卡

### 打卡动作

当用户在景点半径内时，可执行正式打卡。一次打卡记录至少包含：

- 用户 ID
- 景点 ID
- 打卡时间
- 打卡时坐标
- 照片列表（可选）
- 文本备注（可选）

首版默认同一景点只计一次正式打卡进度，但允许后续扩展成多次补充内容。

### 图鉴点亮

图鉴页是首版的主反馈面板。打卡成功后：

- 景点状态变为 `已打卡`
- 总完成度更新
- 首页与“我的”页同步更新

### 收集反馈

首版反馈不复杂，但要明确：

- 已打卡数量 / 总数量
- 完成百分比
- 轻量徽章，例如“初级旅行家”“城市探索者”

徽章只作为增强反馈，不应成为复杂独立系统。

## 数据模型

### ScenicSpot

- `id`
- `name`
- `scopeType`
- `scopeId`
- `lat`
- `lng`
- `coverImage`
- `intro`
- `radiusMeters`
- `badgeName`

表示一个预置旅游打卡点。

### CheckinRecord

- `id`
- `userId`
- `spotId`
- `checkinTime`
- `lat`
- `lng`
- `photoList`
- `note`

表示一次正式打卡记录。

### UserProgress

- `userId`
- `scopeType`
- `scopeId`
- `checkedInCount`
- `badgeList`
- `lastCheckinTime`

用于聚合展示用户在当前城市或景区范围下的收集进度，避免首版数据结构被“城市”字段绑定死。

## 业务规则

1. 同一景点默认只允许一次正式点亮
2. 同一景点的重复到访，首版不重复累计进度
3. 景点必须来自预置景点库
4. 必须满足距离条件后才允许打卡
5. 定位误差需要容错，避免误伤真实到访用户

## 异常与边界处理

### 定位权限未授权

- 首页和图鉴页仍可浏览景点信息
- 详情页不显示“立即打卡”，而是引导用户开启定位权限

### 定位失败或精度过低

- 提示用户重试定位
- 保持页面可浏览，不直接报错中断流程
- 如精度信息可用，可在精度过低时提示“暂时无法确认是否在景点附近”

### 不在打卡范围内

- 清晰展示剩余距离
- 不显示可提交态，避免用户误操作

### 重复打卡

- 如果用户已经完成该景点正式打卡，详情页显示“已打卡”状态
- 首版不重复累计图鉴进度

## 测试与验收

首版验证重点不是覆盖所有旅游场景，而是确认核心闭环稳定。

### 关键验收场景

1. 用户首次进入小程序，能完成定位授权并看到附近景点
2. 用户未到范围内时，详情页不会错误展示“立即打卡”
3. 用户进入景点范围后，可稳定看到可打卡状态
4. 用户完成打卡后，记录被保存，图鉴状态同步刷新
5. 用户再次进入已打卡景点，不会重复累计正式进度

### 建议测试维度

- 不同定位精度下的距离判定
- 定位权限拒绝、关闭、重新授权流程
- 单景区和单城市两种首版范围配置
- 有照片 / 无照片两种打卡提交路径

## 技术建议

### 客户端

优先使用微信小程序原生方案，保持依赖简单、贴合平台能力。若团队已有跨端需求，可备选 `Taro`，但这不是首版必需条件。

### 后端

首版优先推荐微信云开发：

- 便于接入微信登录
- 数据库存储够用
- 图片上传链路简单
- 适合低成本 MVP 验证

如果后续需要跨平台或更复杂的数据运营，再迁移到独立后端。

## 风险与后续扩展

### 首版风险

- GPS 精度不稳定，可能影响打卡成功率
- 景点库需要人工维护，数据质量决定体验
- 大型景区可能单一坐标不足，需要多个打卡点
- 防作弊和平衡易用性之间需要拿捏

### 明确的后续扩展

- 支持用户自建地点
- 扩展到多城市
- 增加分享海报或好友可见
- 引入地图探索视图
- 为单景点支持多次游记/照片补充

## MVP 成功标准

首版上线后，如果用户能够稳定完成下面的闭环，就说明方向成立：

1. 授权定位后能看到附近景点
2. 到达景点附近后能顺利打卡
3. 打卡后记录被保存
4. 图鉴状态和完成度正确更新
5. 用户愿意继续去解锁下一个景点
